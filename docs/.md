Quick summary of notable implementation details / missing bits found

Login service returns { requires_2fa: true, method } when 2FA is required. Controller previously did not forward requires_2fa and method → patched to include them in response body so frontend can present OTP UI. (Patched file: auth.controller.ts:1-300)
Server sets refresh_token and access_token as httpOnly cookies in several places (verify OTP, login when tokens present, refresh-token, verify_login_2fa). Frontend cannot read httpOnly cookies.
Access token is also returned in the response body for most flows (except refresh behavior: refresh sets cookies and returns access_token in body as well).
OTP flows include server-side cooldowns, rate limits and blocking; frontend should display retry/resend timers and handle 429 responses.
All endpoints use JSON body. Some endpoints require either user_email or user_phone (never both); validations enforce this.
Auth Endpoints

For all endpoints below base path = /auth.

Register
Method: POST
Route: /auth/register
Description: Create user (email or phone). Server sends verification OTP (email or phone) after creating user.
Authentication Required: No
Request Headers: Content-Type: application/json
Request Body: see schema in auth.validation.ts:1-220 — required: user_name, user_password, user_area, user_city, user_country, and either user_email OR user_phone.
Example JSON Payload:
{
"user_name":"Jane Doe",
"user_email":"jane@example.com",
"user_password":"strongPassword123",
"user_area":"Dhanmondi",
"user_city":"Dhaka",
"user_country":"Bangladesh"
}
Success Response Example (201):
{
"success": true,
"message": "User registered successfully",
"data": {
"user": {
"_id":"<userId>",
"user_name":"Jane Doe",
"user_email":"jane@example.com",
"user_phone":null,
"user_role":"user",
"user_status":"active",
"email_verified": false,
"phone_verified": false
}
}
}
Error Response Example:
{ "success": false, "message": "Email already registered", "data": null }
Frontend Notes:
After success, show an OTP input screen. Use the contact used (email or phone).
Do not call login yet — wait for OTP verification.
Handle validation errors (400) to show field messages.
Verify OTP (for registration/email/phone verification)
Method: POST
Route: /auth/verify-otp
Description: Verify OTP sent during register (marks email/phone verified). Returns access_token in body and sets refresh_token cookie.
Authentication Required: No
Request Headers: Content-Type: application/json
Request Body: { user_email?: string, user_phone?: string, verify_otp: "123456" } (one of email or phone)
Example JSON Payload:
{ "user_email":"jane@example.com","verify_otp":"123456" }
Success Response Example (200) — note: server sets refresh_token cookie:
{
"success": true,
"message": "OTP verified successfully",
"data": {
"access_token":"<jwt.access>",
"user": { "_id":"<id>", "user_name":"Jane Doe", "user_email":"jane@example.com", "email_verified": true, ... }
}
}
Error Response Example:
{ "success": false, "message": "Invalid OTP", "data": null } or { "success": false, "message": "OTP expired", ... }
Frontend Notes:
On success you get an access_token in body and a httpOnly refresh_token cookie; store access token in memory (or as you prefer) and use it for Authorization header for subsequent protected calls.
If server sets cookies (httpOnly), prefer relying on cookies for refresh flows and use in-memory for access token (do not store JWT in localStorage unless consciously accepted).
After success, redirect to app or call GET /auth/get-me.
Resend OTP
Method: POST
Route: /auth/resend-otp
Description: Resend OTP for registration or verification; obeys cooldown/rate-limits.
Authentication Required: No
Request Headers: Content-Type: application/json
Request Body: { user_email?: string, user_phone?: string }
Example:
{ "user_email":"jane@example.com" }
Success Response Example:
{ "success": true, "message": "OTP resend successfully", "data": { } }
Error Response Example: 429 or 400 messages:
{ "success": false, "message": "Please wait 1m 15s before requesting another OTP" }
Frontend Notes:
If response is 429, display cooldown timer using the server message or track client-side 2-minute cooldown.
Prevent repeated requests while cooldown active.
Login
Method: POST
Route: /auth/login
Description: Authenticate with email/phone + password. If user has 2FA enabled, server will send a 2FA OTP and return requires_2fa: true. Otherwise server returns access_token and sets refresh_token cookie.
Authentication Required: No
Request Headers: Content-Type: application/json
Request Body: { user_email?: string, user_phone?: string, user_password: string }
Example:
{ "user_email":"jane@example.com", "user_password":"strongPassword123" }
Success Response Examples:
Case A — credentials valid and 2FA disabled:
Server sets httpOnly cookies (refresh_token, access_token) and returns body:
{
"success": true,
"message": "Login successful",
"data": {
"requires_2fa": false,
"access_token":"<jwt.access>",
"user": { "_id":"...", "user_name":"Jane", "user_email":"jane@example.com", ... }
}
}
Case B — credentials valid but 2FA enabled:
{
"success": true,
"message": "2FA OTP sent successfully",
"data": {
"requires_2fa": true,
"method":"email" // or "phone"
}
}
Error Response Example:
{ "success": false, "message": "Invalid credentials", "data": null } or 401 for unverified email/phone.
Frontend Notes:
After login success with requires_2fa: true, show OTP screen; do NOT call get-me yet.
If requires_2fa: false, store access_token in memory and use it for Authorization header; server also sets refresh_token cookie for refresh flows.
Include Authorization: Bearer <access_token> for protected calls, OR rely on cookies if server and client are same-site and you use cookie-based auth.

Authentication Required: Yes (use Authorization: Bearer <access_token> header OR httpOnly access_token cookie).
Request Headers:
Preferred: Authorization: Bearer <access_token> (if frontend stores access token)

Authentication Required: Yes
Request Headers: Authorization: Bearer <access_token> or cookie + Content-Type: application/json
Request Body: { "two_factor_otp_method":"email" }
Example:
{ "two_factor_otp_method":"email" }
Success Response Example:
{
"success": true,
"message":"2FA enable OTP sent successfully",

Authentication Required: Yes
Request Headers: Content-Type: application/json + auth
Example Enable:

{ "enabled": false }
Success Response Example:
{ "success": true, "message": "Two-factor authentication enabled successfully" }
Error Response Example:
{ "success": false, "message": "OTP is required to enable two-factor authentication" }
Frontend Notes:
After enabling, call GET /auth/get-me to refresh user profile.
If enabling, disable UI until confirmation. If disabling, you can proceed immediately after success.
Change Password — Request
Method: POST
Route: /auth/change-password-request
Description: Verify old password; if 2FA enabled it triggers a 2FA OTP; otherwise returns that 2FA is not required and client can call confirm endpoint immediately.
Authentication Required: Yes
Request Headers: Content-Type: application/json + auth
Request Body: { "old_password":"oldPass123" }
Success Response Example:
If 2FA disabled:
{ "success": true, "message":"Password change request successful", "data": { "require_2fa": false } }
If 2FA enabled: { "success": true, "message":"Password change request successful", "data": { "require_2fa": true } } and server sends OTP
Error Response Example:
{ "success": false, "message":"Invalid old password" }
Frontend Notes:
If require_2fa: true, show OTP input and then call confirm endpoint.
If false, show "set new password" UI and call confirm endpoint immediately.
Confirm Password Change
Method: POST
Route: /auth/confirm-password-change
Description: Set new password. If 2FA required, include verify_otp.
Authentication Required: Yes
Request Body: { "new_password":"NewPass123", "verify_otp":"123456" } (verify_otp optional if 2FA disabled)
Success Response Example:
{ "success": true, "message":"Password changed successfully", "data": { ... } }

Refresh Token
Method: POST
Route: /auth/refresh-token
Description: Exchange refresh token (cookie or in-body) for new access + refresh tokens. Server sets new cookies and returns access_token in body and user.
Authentication Required: No (but requires a refresh token)
Request Headers: Content-Type: application/json (if sending refresh_token in body)
Request Body: { "refresh_token":"..." } optional — server will use refresh_token cookie if present.
Success Response Example:
{
"success": true,
"message":"Token refreshed successfully",
"data": {
"access_token":"<jwt.access>",
"user": { ... }
}
}
Error Response Example:
{ "success": false, "message":"Invalid refresh token" } (401)
Frontend Notes:
Usually call when GET /auth/get-me or other protected route returns 401.
Use fetch with credentials: 'include' to send cookies. If you store refresh token client-side (not recommended), include it in the body.
After successful refresh, update in-memory access_token and retry the original request.
Flows — Step-by-step (what to call, expected response, next call)

New user sign-up (email flow example)
Call: POST /auth/register with email + password + required fields.
Expect: 201 with user object; server sends OTP to email.
Next: Show OTP screen and call POST /auth/verify-otp with user_email + verify_otp.
On verify success: server returns access_token and sets refresh_token cookie. Store access_token in-memory and call GET /auth/get-me.
Login (no 2FA)
Call: POST /auth/login.
If requires_2fa:false → server returns access_token & sets refresh_token cookie. Store access token in memory and call GET /auth/get-me.
If requires_2fa:true → server returns { requires_2fa: true, method }. Show OTP screen and call POST /auth/verify-login-2fa. On success server returns access_token and sets cookies.
Enable 2FA
Call POST /auth/enable-2fa with selected method → server sends OTP to verified contact and returns method.
Show OTP screen → call POST /auth/toggle-2fa with { enabled:true, verify_otp } to confirm.
Change password
Call POST /auth/change-password-request with old_password.
If require_2fa:true → server sent OTP; display OTP input then call POST /auth/confirm-password-change with { new_password, verify_otp }. If require_2fa:false → call POST /auth/confirm-password-change with { new_password }.
Forgot/reset password
Call POST /auth/forgot-password with user_email or user_phone → server sends OTP.
Call POST /auth/reset-password with user_email/user_phone, verify_otp, new_password.
Logout
Call POST /auth/logout → server clears cookies. Clear client state.
Token refresh / protected routes
On any protected request returning 401:
Try POST /auth/refresh-token (cookies preferred). On success, update access token in-memory and retry the original request. If refresh fails → redirect to login.
When OTP screen should appear

Immediately after POST /auth/register (server sends registration OTP).
After POST /auth/login if response data.requires_2fa === true.
After POST /auth/enable-2fa (to confirm enabling).
After POST /auth/change-password-request when server responds with require_2fa: true.
After POST /auth/forgot-password (to reset password).
After POST /auth/change-contact-request (to confirm new contact).
Token storage & usage recommendations (practical)

Refresh token: Server sets httpOnly refresh_token cookie (recommended). Do NOT store refresh token in localStorage. Use cookie-based refresh.
Access token: Server sets httpOnly access_token cookie too in some responses. However, an access_token is also returned in response body. Recommended frontend pattern:
Store access_token in memory (React context or Redux). Avoid long-term storage unless needed.
For server-set httpOnly cookies, prefer using credentials: 'include' on fetch so cookies are sent automatically.
For cross-site deployments (Next.js on different domain), ensure cookies are set with sameSite: none and secure: true (server already handles this for production).
Authorization header: If you keep access_token in memory, send Authorization: Bearer <access_token> for API calls; otherwise rely on cookies (include credentials).
How protected routes work

Middleware check_auth accepts access token either from:
Authorization: Bearer <access_token> header OR
access_token cookie (httpOnly).
check_auth validates token and loads user; it checks password_changed_at to invalidate old tokens (and is_deleted/user_status).
If 401 returned, try POST /auth/refresh-token to obtain new tokens and retry.
Common error cases & frontend handling

400 Bad Request — invalid payload / validation errors: show field-level messages.
401 Unauthorized — missing/invalid access token: attempt refresh; if refresh fails navigate to login.
403 Forbidden — account banned/deactivated: show account-level message and prevent access.
429 Too Many Requests (OTP cooldown & blocking): show the server message and disable resend until cooldown finishes.
OTP errors:
"Invalid OTP": show one-time error and allow retry.
"OTP expired": prompt user to request resend (call resend-otp).
"OTP blocked" / too many attempts: block for a minute(s) as server message; show instructions and cooling timer.
Password-change / reset errors: New password must be different from current password — show appropriate validation message.
Implementation tips for a mid-level React/Next.js dev

Use a central auth provider (React context) that holds in-memory access_token and user object.
On app start:
If SSR/Next.js: try POST /auth/refresh-token server-side (with credentials) to hydrate user.
If SPA: call GET /auth/get-me (sending Authorization if you have access token or credentials: 'include') and fallback to POST /auth/refresh-token on 401.
OTP flow UI:
After any OTP-triggering response, show OTP input with countdown/resend UI based on messages.
When receiving requires_2fa:true from login, store a short-lived client flag awaiting_2fa_for=<email|phone> so UI knows to call verify-login-2fa.
Resend button:
Disable for at least 2 minutes client-side (server enforces 2 min cooldown); show server messages for exact remaining time.
Token refresh:
Implement automatic refresh when access token expires OR on 401: call POST /auth/refresh-token (with credentials: 'include'), update in-memory tokens, then retry the original request once.
Cookie vs header:
If you rely on cookies, set fetch/axios to send credentials: fetch(url,{ credentials: 'include', ... }) or axios.defaults.withCredentials = true.
For Authorization header approach, keep access_token in memory and send Authorization header.
